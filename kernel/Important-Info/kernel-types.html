Kernel architecture types

Different operating systems organize their kernel services in different ways. Each style trades off simplicity, speed, safety, and flexibility.

Monolithic kernel

A monolithic kernel puts most of the operating system in one large privileged body of code. This makes it very fast and straightforward for many early kernels, but it can become hard to maintain because the kernel grows into a big, tightly coupled system.

Microkernel

A microkernel keeps only the most essential pieces in the kernel: scheduling, IPC, and basic memory management. Other services like filesystems, drivers, and networking usually run as separate user-space processes, which improves modularity and fault isolation.

Nanokernel

A nanokernel goes even further than a microkernel and strips the kernel down to the bare minimum needed to manage hardware and basic abstractions. It is extremely minimal and usually only makes sense for highly specialized or research-oriented systems.

Exokernel

An exokernel gives applications much more direct control over hardware resources. Instead of hiding everything behind a heavy abstraction layer, it exposes low-level mechanisms and lets the system or app decide how to use them. This approach is flexible and efficient, but it is also more complex to build and use correctly.

Hybrid kernel

A hybrid kernel tries to combine the strengths of both monolithic and microkernel designs. It keeps some services in the kernel for performance, while still separating others into modules or user-space components. This is common in practical systems where speed matters but some modularity is still desired.

Unikernel

A unikernel is a single-purpose image that bundles the application and the necessary operating system support into one very small artifact. It is often used in cloud, embedded, or specialized environments where minimal size and predictable behavior matter more than broad general-purpose features.

Full ring zero OS

A full ring zero OS runs nearly everything at privilege level 0, meaning the kernel and its services share the same highest-privilege mode. This is simple and fast in a traditional sense, but it gives up some of the protection benefits of separating user and kernel modes.

Where HepOS fits

HepOS is best described as an exokernel-inspired project with a growing kernel-space libOS.

Its kernel focuses on multiplexing hardware directly, while higher-level desktop and filesystem services are built above that core. That makes it a good fit for the exokernel style, even though it still has many monolithic-era conveniences during development.
